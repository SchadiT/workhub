Modul 03 · Claude Mastery Curriculum

Projects richtig nutzen.

Wann brauchst du ein eigenes Projekt, wann reicht ein Chat? Was gehört in Instructions, was in Files, was in Memory? Und warum ist deine aktuelle Trennung (Turbo vs. Mastery) genau richtig.

Reading: 45 Min Hands-on: 30 Min Schwierigkeit: Einsteiger
Lernziele

📋 Was du nach diesem Modul kannst

Ziele
  • Entscheiden ob eine Aufgabe ein Projekt oder einen normalen Chat braucht
  • Strukturieren ein neues Projekt: Instructions, Files, Memory richtig befüllen
  • Verstehen warum du aktuell zwei Projekte hast — und warum das richtig ist
  • Verbessern deine bestehenden Projekt-Instructions für bessere Ergebnisse
  • Erkennen wann ein drittes Projekt Sinn macht (und wann nicht)
Sektion 1

🧠 Mental Model — Projekt vs. Chat vs. Raum

Denk so: Ein Claude-Projekt ist wie ein dedizierter Arbeitsraum — mit eigenen Regeln an der Wand, eigenen Files auf dem Schreibtisch, und einem Assistenten der genau weiß was in diesem Raum passiert.

Ein normaler Chat ist wie ein Meeting im Flur. Schnell, pragmatisch, aber ohne persistenten Kontext und ohne dedizierte Files.

OHNE PROJEKT: ┌────────────────────────────────────────────────────────┐ │ Normaler Chat │ │ │ │ Du: "Ich bin Schadi, CEO Grelle Forelle, hier ist │ │ was du über meinen Hub wissen musst: ..." │ │ [15 Minuten Kontext setzen] │ │ [Ende des Chats → alles weg] │ │ [Nächster Chat → alles neu erklären] │ └────────────────────────────────────────────────────────┘ MIT PROJEKT: ┌────────────────────────────────────────────────────────┐ │ TURBO PROJECTS PROJEKT │ │ ─────────────────────────────────────────── │ │ Instructions: [Wer du bist, wie Grelle Forelle │ │ funktioniert, Kommunikations-Regeln] │ │ Files: [SKILL.md, Hub-Code, Business-Context] │ │ Memory: [Auto-extrahierter Kontext] │ │ │ │ Neuer Chat im Projekt: │ │ Du: "Bau mir ein Factsheet für Samstag" │ │ Claude: [weiß sofort alles] → direkt loslegen │ └────────────────────────────────────────────────────────┘
💡 Core Insight

Ein Projekt spart dir bei jedem neuen Chat das Kontext-Setup. Bei 5 Chats pro Tag × 5 Minuten Setup = 25 Minuten täglich verschwendet. Mit Projekt: 0 Minuten.

Sektion 2

🎯 Wann brauchst du ein Projekt?

✅
Projekt: JA
Wenn du einen wiederkehrenden Workflow hast der immer denselben Kontext braucht. Wenn du Files hast die Claude regelmäßig referenzieren soll. Wenn du spezifische Verhaltens-Regeln für diesen Kontext brauchst die sich von anderen unterscheiden.
❌
Projekt: NEIN
Wenn es eine Einzel-Task ist die nie wieder vorkommt. Wenn kein persistenter Kontext nötig ist. Wenn Global Memory + User Preferences ausreichen um den gewünschten Stil zu bekommen.
🤔
Projekt: VIELLEICHT
Wenn du 3+ Mal in Folge den gleichen Kontext erklärt hast — dann ist es Zeit für ein Projekt. Wenn du einen neuen Workflow aufbaust der sich wiederholen wird.

Die Entscheidungs-Frage

"Werde ich das mehr als 5 Mal brauchen, mit ähnlichem Kontext?"

Ja → Projekt. Nein → normaler Chat.

Sektion 3

🏗️ Aufbau eines Projekts — Die 3 Elemente

Element 1: Project Instructions

Das ist der System-Prompt des Projekts. Wird in jeden Chat im Projekt geladen. Hier definierst du:

  • Zweck des Projekts — Was passiert hier, was passiert nicht
  • Kommunikations-Regeln — Ton, Format, Sprache
  • Kontext — Wer du bist in diesem Kontext, relevante Personen, Systeme
  • Verhaltens-Regeln — Was Claude tun und lassen soll

Element 2: Project Files

Dateien die Claude referenzieren soll. Read-only. Claude liest sie, ändert sie nicht. Gut für:

  • Skill-Definitionen (SKILL.md Files)
  • Templates und Locked-Formats
  • Business-Context Dokumente
  • Referenz-Daten (Preislisten, Kontakte, etc.)
  • Fertige Module / Lernmaterialien
⚠️ File-Limit

Es gibt ein Größen-Limit für Project Files. Halte Files kompakt und relevant. Nicht alles reinladen — nur was Claude wirklich braucht.

Element 3: Project Memory

Was Claude automatisch aus Gesprächen in diesem Projekt extrahiert. Du kannst es in Settings → Memory filtern nach Projekt. Manuell hinzufügen: im Chat direkt sagen "Merke dir für dieses Projekt..."

EIN PROJEKT — DREI ELEMENTE: ┌──────────────────────────────────────────────────────┐ │ PROJECT INSTRUCTIONS │ │ "Zweck: Hub-Dev. Kein operative ToDos. ..." │ │ → geladen in JEDEN Chat in diesem Projekt │ ├──────────────────────────────────────────────────────┤ │ PROJECT FILES │ │ create-factsheet/SKILL.md (5 KB) │ │ create-security-order/SKILL.md (4 KB) │ │ Business_Context_Lite.md (3 KB) │ │ → Claude kann durchsuchen + referenzieren │ ├──────────────────────────────────────────────────────┤ │ PROJECT MEMORY │ │ "Hub hat 8 Hauptsektionen" │ │ "LOCKED Format = v15.9.165" │ │ → auto-extrahiert + manuell gesetzt │ └──────────────────────────────────────────────────────┘ │ ▼ NEUER CHAT im Projekt: Alle drei Elemente sofort verfügbar ✅
Sektion 4

📁 Deine aktuellen Projekte — Warum die Trennung richtig ist

Du hast aktuell zwei Projekte: Turbo Projects und Claude Mastery. Das ist eine bewusste Entscheidung die gut funktioniert.

Ohne Trennung — Was schief geht
Alles in einem Projekt: Hub-Dev + Lernmaterial + Daily Ops
Problem: Claude bekommt conflicting Instructions. "Baue den Hub" vs "Erkläre mir Konzepte" vs "Mach Daily Briefing" → Instructions werden zu lang und verwässert → Claude weiß nicht welche Regel gerade gilt → Format-Fehler, falsche Tonalität, verwischte Grenzen → Du musst mehr erklären, nicht weniger
Mit Trennung — Was richtig läuft
Turbo Projects: Hub-Dev, Daily Ops, MCPs, Skills
Claude weiß: "Hier baue ich, hier helfe ich operativ, hier nutze ich MCP-Tools" → Präzise Instructions, klarer Scope → Kein Kontext-Bleeding zwischen Lernmaterial und Produktivarbeit → Jeder Chat startet mit dem richtigen Mind-Set

Wann solltest du ein drittes Projekt erstellen?

Wenn ein neuer, dauerhafter Workflow entsteht der eigenen Kontext, eigene Files, und andere Verhaltens-Regeln braucht als die bestehenden Projekte.

Mögliche zukünftige Projekte für dich:

  • Email-Projekt — Wenn du systematisch Promoter-Mails / Sandra-Mails / Team-Kommunikation mit Claude managst
  • Analytics-Projekt — Wenn du tief in MAIN DB Analysen gehst mit eigenem Kontext
  • Automation-Projekt — Wenn n8n + Zapier-Workflows einen eigenen Kontext-Raum brauchen
💡 Regel

Starte kein neues Projekt bevor du nicht 5+ Mal denselben Kontext in einem normalen Chat erklärt hast. Dann weißt du genau was rein muss.

Sektion 5

📝 Gute Project Instructions schreiben

Struktur einer guten Instruction

Deine aktuellen Instructions (das was am Anfang dieses Chats stand) sind ein gutes Beispiel. Hier das Prinzip:

GUTE INSTRUCTIONS STRUKTUR: 1. ZWECK (1-3 Sätze) Was passiert in diesem Projekt? Was passiert NICHT in diesem Projekt? → Setzt Scope sofort 2. KONTEXT (3-5 Zeilen) Wer bist du in diesem Kontext? Welche Personen/Systeme sind relevant? → Claude braucht keine Erklärungen mehr 3. VERHALTENS-REGELN (Liste) Was soll Claude tun? Was soll Claude NICHT tun? → Verhindert häufige Fehler 4. FORMAT-REGELN (wenn nötig) Wie sollen Outputs aussehen? Welche Formate sind LOCKED? → Konsistenz ohne Wiederholung 5. WORKFLOW (optional) Plan-First? Build-Go Signale? Session-Regeln? → Prozess-Effizienz
Beispiel · Schwache vs. Starke Instructions
SCHWACH: "Du hilfst mir mit meinem Club-Geschäft in Wien. Sei professionell und direkt."
STARK: "Dies ist das Turbo Projects Projekt für Grelle Forelle (Club, Wien). HIER: Hub-Dev, Daily Ops, MCPs, Skills. NICHT HIER: Lernmaterial, akademische Erklärungen. Kontext: Schadi = CEO, ~30 Staff, 5 Departments. Regeln: Du-Form, kein Fluff, LOCKED Formats nie ändern. Build-Go: 'mach'/'ok'/'go' = direkt ausführen. Token-Count am Ende jeder Antwort."
Sektion 6

🚨 Anti-Patterns

Anti-Pattern #1
Ein Projekt für alles

Du packst Hub-Dev, Lernmaterial, Daily Ops und Email-Drafts in ein Projekt. Instructions werden lang, widersprüchlich, ineffektiv. Trenne by function — nicht by convenience.

Anti-Pattern #2
Instructions zu vage

"Sei ein hilfreicher Assistent für mein Business." Claude kennt dein Business nicht, kennt die Rules nicht, kennt den Scope nicht. Instructions müssen spezifisch sein — Scope, Kontext, Regeln, Signale.

Anti-Pattern #3
Zu viele Files hochladen

Du lädst alles in Project Files — jedes Email-Template, jeden Event-Plan, jede alte Version. Das erhöht Konfusion. Nur was Claude wirklich regelmäßig braucht: Skills, Locked-Formats, Business-Context.

Anti-Pattern #4
Projekt nie updaten

Dein Workflow ändert sich — neue Skills, neue MCPs, neue Personen im Team — aber Instructions bleiben veraltet. Quartals-Review: Instructions updaten, veraltete Files ersetzen.
